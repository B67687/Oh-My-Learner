## Key Design Decisions (as-built)

### Decision 1: Scheduler interface with dual backends (SM-2 + FSRS-5)

`core/scheduler.go` defines a `Scheduler` interface. `SM2Scheduler` is the active backend. `FSRSScheduler` (in `core/scheduler_fsrs.go`) implements the FSRS-5 algorithm with the canonical 22-parameter set. Both share the same `ReviewCard(CardState, ReviewQuality) CardState` signature.

The package-level `ReviewCard` convenience function delegates to `SM2Scheduler` by default. The review command (`cmd/review_loop.go`) also instantiates `SM2Scheduler` explicitly. FSRS is structurally complete but not yet selectable at runtime — this matches the v3 spec's stated intent ("FSRS swap-ready").

**Known limitation:** FSRS state (`Stability`, `Difficulty`) lives on `CardState` but is not persisted to the `cards` table. Every FSRS review currently re-initializes from `S₀`/`D₀`. To make FSRS fully operational, the database schema and `InsertCard`/`UpdateCardState` methods need `stability` and `difficulty` columns.

### Decision 2: FSRS-5 algorithm implementation

`core/scheduler_fsrs.go` implements FSRS-5 from the published specification:

- **22 parameters** (`fsrsW`) — the canonical set from Jarrett Ye's FSRS-5 paper
- **4 grades** — again (lapse), hard, good, easy, mapped from SM-2's 0-5 quality scale
- **Core formulas**: `retrievability()`, `stabilityAfterRecall()`, `stabilityAfterForget()`, `difficultyAfterRecall()`, `difficultyAfterForget()`
- **Interval**: `I = 9·S·(1/r − 1)` which simplifies to `I = S` at `r_target = 0.9`
- **Caps**: max interval 365 days, stability floored at `w[0]`, difficulty clamped to [1, 10]

7 dedicated FSRS tests cover: grade mapping, retrievability, stability growth/decay, difficulty clamping, new-card init, hard penalty, max interval, multi-review monotonicity, and compile-time interface satisfaction.

### Decision 3: Selective interleaving by knowledge type

Cards carry a `knowledge_type` field (`declarative` or `procedural`). The review session in `cmd/review.go` separates due cards by type:

- **Procedural** cards (algorithms, code, problem-solving) are shuffled and interleaved.
- **Declarative** cards (definitions, facts) are blocked by subject — subjects are shuffled, but cards within a subject are presented together.

This matches research finding that interleaving hurts declarative fact recall (g=-0.39) while benefiting procedural skills (g=+0.42).

### Decision 4: Self-explain with storage and retrieval

Unlike v2 where self-explain responses were prompted but discarded, v3 stores them:

- `reviews` table has a `self_explain_response` column (migration in `storage.go:113`).
- `InsertReview(cardID, quality, selfExplainResponse)` persists the response.
- `learn report --verbose` queries and displays recent self-explanations.
- Base `learn report` shows only streak, retention, and daily activity.

### Decision 5: Streak tracking with backlog forgiveness

`core/storage_meta.go` implements a `streak` table with a singleton row:

- Consecutive review days increment the streak.
- 1-2 missed days preserve the streak (no increment, no reset).
- 3+ missed days reset the streak to 1.
- Backlog forgiveness: if more than `daily_review_limit` (default 50) cards are due, the session randomly selects that many and presents them. No compound punishment.

### Decision 6: AI generation via DeepSeek V4 Flash (free-tier)

`agent/agent.go` implements the `Agent` interface using DeepSeek's OpenAI-compatible API. The agent constructs a system prompt specifying the 4 template types and `knowledge_type` field, sends it with the user's topic, and parses the JSON response. Response parsing tolerates markdown code block wrappers and validates each card's type, knowledge_type, question, and answer fields before returning.

No changes from v2. (6 test fixtures + 6 tests remain.)

### Decision 7: Dual card source — AI + TOML packs

Both paths coexist:

- `learn add --ai <topic>` — generates cards via DeepSeek.
- `learn add <subject>` — installs from `subjects/<subject>.toml` (legacy pack system).

### Decision 8: Adherence infrastructure (hooks)

`learn hook` generates implementation-intention hooks:

- `--shell bash|zsh`: installs a shell prompt snippet showing due card count in `RPROMPT` (zsh) or `PS1` (bash).
- `--tmux`: installs a tmux status-right integration showing due count.
- Hooks are written to `~/.config/oh-my-learner/hooks/`.

### Decision 9: Pure Go SQLite (no CGO)

`modernc.org/sqlite` — no native dependencies. Single binary distribution.

### Decision 10: Storage split to 4 files

To respect the 250/40 LOC rule, `storage.go` was split into:

- `storage.go` — connection, migration, schema (130 lines)
- `storage_cards.go` — card CRUD, review insertion (194 lines)
- `storage_meta.go` — streak, daily log, weekly retention (125 lines)
- `storage_subjects.go` — subject CRUD, prerequisites, due counts (167 lines)

### Decision 11: Review split to 2 files

`review.go` — session orchestration (shuffle, backlog cap, type split) (150 lines)
`review_loop.go` — per-card rendering, quality collection, scheduler call (138 lines)

## Data Flow

### TOML pack path

```
learn add <subject> → reads subjects/<subject>.toml
                  → parses Template + SubjectMeta via core.LoadSubjectPack()
                  → generates Card instances with DefaultCardState()
                  → stores in SQLite via core.InsertCard()
```

### AI generation path

```
learn add --ai <topic> → agent.NewDeepSeekAgent() reads OML_DEEPSEEK_KEY
                      → agent.GenerateCards(topic)
                      → constructs system prompt with 4 types + knowledge_type
                      → POST to DeepSeek API (deepseek-chat model)
                      → parses JSON response, validates each card
                      → converts AIGeneratedCard → core.Template
                      → stores in SQLite via core.InsertCard()
```

### Review path

```
learn review → queries DueCards() from SQLite (ORDER BY RANDOM())
            → applies backlog forgiveness cap (default 50 cards)
            → separates by knowledge_type:
                procedural → shuffled (interleaved)
                declarative → blocked by subject (subjects shuffled)
            → for each card:
                → GetCardWithTemplate() → RenderTemplate() → variable substitution
                → display type-specific UI (code block, Q&A, etc.)
                → [press Enter to reveal answer]
                → self-explanation prompt (skipped in --mode speed)
                → collect quality rating (0-5)
                → SM2Scheduler.ReviewCard() → UpdateCardState() + InsertReview(..., selfExplain)
            → UpdateStreak(today) — 2-day forgiveness
            → LogDailyActivity(today, reviewed, recalled)
            → display session summary + streak info
```

### Hook integration

```
learn hook --shell zsh → writes oh-my-learner-prompt.zsh to hooks dir
                      → user sources it in ~/.zshrc
                      → shell prompt shows due card count on each prompt

learn hook --tmux → writes oh-my-learner-tmux.conf to hooks dir
                 → user sources in ~/.tmux.conf
                 → tmux status-right shows due card count
```

## Database Schema (as-built)

```
subjects:         id TEXT PK, name TEXT
cards:            id TEXT PK, subject_id, template_type, knowledge_type,
                  template_question, template_answer, variables (JSON),
                  easiness_factor, interval_days, repetition,
                  next_review_at, created_at
                  -- NOTE: stability/difficulty not stored (FSRS limitation)
reviews:          id TEXT PK, card_id, quality, reviewed_at, self_explain_response
subject_prerequisites: (subject_id, prerequisite_id) PK
streak:           id INTEGER PK CHECK(id=1), current_streak, longest_streak, last_review_date
daily_log:        date TEXT PK, cards_reviewed, cards_recalled
```

Indexes: `cards(subject_id)`, `cards(next_review_at)`, `reviews(card_id)`, `subject_prerequisites(subject_id)`
