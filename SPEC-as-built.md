## Key Design Decisions (as-built)

### Decision 1: SM-2 with session-level interleaving display
Cards are scheduled independently via SM-2 (EF+interval). The review session displays cards with subject-mixing hints but does NOT force random ordering within the session. Cards appear in `DueCards` query order (which approximates due time). The UX displays subject counts and warns if cards are from a single subject.

### Decision 2: Template-based generation (4 types)
Standard Q&A, code-trace, debug-find, explain-why. All 4 types are rendered with type-specific display in the review loop. Variables are substituted via Go templates.

### Decision 3: Pure Go SQLite (no CGO)
`modernc.org/sqlite` ‚Äî no native dependencies. Single binary distribution.

### Decision 4: Session UX tips
The review command displays contextual tips: encourages adding more subjects when fewer than 10 cards are due, encourages interleaving when all cards are from one subject.
